## Goal
Make authorization and engagement scope a durable, mechanically enforced input to every autonomous investigation, not a prompt disclaimer or an assumption inherited from the operator's environment.

## Requirements
- [ ] Define a versioned scope envelope covering owned target hashes/paths, permitted hosts/domains/IP ranges, accounts, time window, action classes, network egress, credential use, data handling, and prohibited actions
- [ ] Bind each run and campaign to an immutable scope revision and record who approved it, when, and why
- [ ] Evaluate scope before scheduling and again at every external/deferred tool boundary so model output cannot bypass it
- [ ] Treat account registration, challenge enrollment, credential creation, submission, persistence, destructive actions, third-party messaging, and scope expansion as separate explicit authorities
- [ ] Default network scope to none and external account creation/submission to forbidden
- [ ] Map the scope envelope into remote-worker network policy and future range egress controls without handing workers controller credentials
- [ ] Require a durable operator decision for any scope revision; approvals bind to exact content and expire with the configured window
- [ ] Redact secrets and sensitive target identifiers from browser/notification projections according to policy while retaining auditable opaque references
- [ ] Preserve denial reasons and attempted out-of-scope actions as security events

## Acceptance Criteria
- [ ] A prompt-injected worker request to contact an unlisted host is denied before network execution and cannot self-authorize a revision
- [ ] A listed offline fixture works under network-none policy; an allowed lab service works only on its exact approved endpoint
- [ ] Expired, modified, missing, or mismatched scope records fail closed across controller restart and remote dispatch
- [ ] Registering for or submitting to a live challenge remains forbidden unless separately and explicitly authorized
- [ ] Scope decisions are auditable without persisting raw credentials or private paths in browser-facing records
- [ ] Tests distinguish tool safety permission from target/action authorization; satisfying one never implies the other

## Plan
Start with a pure policy model and decision matrix, integrate at run creation and Rekit dispatch, then remote transport and campaign enforcement. Add a migration/default policy for existing local read-only runs.

## Notes
Isolation is not authorization, and a user-approved gated tool is not permission to use it against arbitrary targets. This task deliberately closes the gap exposed by autonomous CTF registration and open-ended real-software research in the reviewed workflow.
