## Goal
Implement the existing durable-project-memory design so a new process, worker, model, or harness can continue an investigation without rereading an original conversation or repeating known failures.

## Requirements
- [ ] Add an append-only, versioned project memory stream with monotonic sequence numbers and replay-safe unknown event handling
- [ ] Represent goal history and pivots, workstreams, attempts, decisions, research questions, theories, next actions, and bounded session compactions with stable IDs
- [ ] Reference ledger events, artifacts, run events, questions, capability leads, and external sources instead of copying bulky evidence
- [ ] Preserve failed attempts, rejected theories, alternatives, rationale, reconsideration triggers, blockers, and stop conditions
- [ ] Provide a pure deterministic fold into a typed ProjectMemory snapshot
- [ ] Provide a validated narrow write API; models return structured proposed memory actions and cannot append arbitrary events
- [ ] Build a token/size-bounded memory_context projection containing only the current goal, active work, relevant prior failures, open questions, supported/testing theories, and highest-priority unblocked actions
- [ ] Define compaction so summaries cite retained evidence and never erase unresolved uncertainty or negative results
- [ ] Add deterministic Markdown export as a projection, never a co-equal writable source of truth

## Acceptance Criteria
- [ ] Replaying the same stream reconstructs byte-equivalent folded state across process restarts
- [ ] A second harness resumes the next action correctly without access to the original model transcript
- [ ] A previously failed method is present in bounded context when it would otherwise be proposed again
- [ ] Oversized histories remain within the configured context budget while every included claim retains resolvable provenance
- [ ] Duplicate structured actions are idempotent and concurrent writers cannot reorder or overwrite history
- [ ] Exporting unchanged state twice produces identical Markdown; importing or editing that Markdown cannot mutate runtime state
- [ ] Corrupt tails, unknown future event types, and missing optional evidence produce explicit degraded state rather than invented facts

## Plan
Land the protocol, append API, fold, and replay tests first. Integrate validated harness actions and context construction second. Add export and Mission Control read-model fields only after canonical replay is stable.

## Notes
Reuse ledger.jsonl/run storage and inbox state by reference. Do not create a second artifact, run, or human-decision database. Raw chain-of-thought and unlimited transcripts are not project memory.
