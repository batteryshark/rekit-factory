## Goal
Capture enough material evidence to reproduce and audit research without turning raw terminal transcripts, screenshots, model chatter, secrets, and duplicate frames into the product or permanent knowledge.

## Requirements
- [x] Establish semantic events and structured tool results as the default evidence path
- [x] Capture terminal output as bounded, content-addressed artifacts only when required by a test or proof policy; preserve exact bytes separately from redacted display excerpts
- [x] Capture screenshots only for GUI/visual state that cannot be represented reliably as structured data
- [x] Deduplicate screenshots and output chunks by content/perceptual hash, record capture reason and source, and reject meaningless repeated frames
- [x] Record timestamps, tool/worker/invocation IDs, environment identity, target hash, media type, and capture policy on every retained artifact
- [x] Add deterministic secret/private-key/token screening plus pluggable sensitive-data classification before browser, notification, export, or knowledge-candidate use
- [x] Define per-run byte/count budgets, truncation semantics, retention classes, citation pinning, expiration, and operator hold/delete controls
- [x] Never treat provider chain-of-thought as required proof; retain only the already-supported provider state needed for continuation and bounded operator diagnostics
- [x] Preserve an audit event when material is withheld, redacted, truncated, deduplicated, expired, or quarantined

## Acceptance Criteria
- [x] A CLI-only fixture produces no screenshots and retains only the output needed by its proof
- [x] A GUI fixture retains one meaningful frame while duplicates and irrelevant full-desktop captures are excluded
- [x] Secrets present in raw output never appear in Mission Control, notifications, dossiers, or knowledge candidates
- [x] Evidence cited by an accepted finding cannot expire silently; retention conflicts become an operator-visible state
- [x] Capture limits remain enforced across restart and concurrent workers
- [x] Hash, provenance, redaction, and deletion/retention behavior have deterministic tests with no live sensitive material

## Plan
Define capture and retention policy records first, then terminal/tool artifact integration, then visual capture adapters, redaction projections, and lifecycle cleanup. Keep raw and redacted representations explicitly related but independently hashed.

## Notes
More screenshots are not better evidence. The system should prefer structured observations, exact reproducible commands, and small artifacts; screenshots are a selective fallback for human-observable GUI state.
