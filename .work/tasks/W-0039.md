## Goal
Make every Rekit capability declare the semantic authorities it may exercise so Factory can enforce W-0022 scope from a versioned catalog contract instead of inferring consequences from network and input-execution flags or conflating authorization with safety tier.

## Requirements
- [ ] Extend the compatible Rekit manifest schema with explicit action authorities such as read, execute, modify, network, credential use, account creation, challenge enrollment/submission, persistence, destructive change, and third-party messaging
- [ ] Keep safety tier, isolation needs, and operator tool consent separate from target/action authorization metadata
- [ ] Define conservative compatibility behavior for legacy manifests; ambiguity must narrow availability or require review rather than grant authority
- [ ] Validate declared authorities against network mode, target mutation, credential/account inputs, dispatcher behavior, and any prohibited combinations
- [ ] Carry the declarations through public/private federated roots without leaking private root paths or credentials
- [ ] Make Factory derive creation and dispatch scope checks from the declaration plus exact runtime intent; model arguments cannot weaken the manifest floor
- [ ] Version and hash the effective manifest so approvals and proof records can cite the exact authority contract used
- [ ] Migrate public Rekit and compatible `rekit-private` manifests with reviewable, least-authority declarations

## Acceptance Criteria
- [ ] A read-only offline capability works under the minimum exact-target grant
- [ ] A mutating, networked, credential-using, account-creating, or submitting fixture is denied unless every separately declared authority and exact runtime intent is approved
- [ ] A low safety tier cannot bypass missing semantic authority, and operator consent cannot widen target scope
- [ ] Unknown, malformed, contradictory, or omitted high-impact declarations fail closed with an auditable reason
- [ ] Federated public/private capability projections expose safe authority names and content identity without source paths or secret values
- [ ] Existing manifest compatibility is deterministic and covered by migration tests rather than an undocumented inference

## Plan
Version the shared registry contract and validator first, migrate representative public/private tools, then replace Factory's compatibility inference with declaration-driven scope requests while retaining a fail-closed legacy bridge.
