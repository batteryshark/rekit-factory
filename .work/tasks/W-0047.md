## Goal
Add a genuine incremental `factory-outcomes/v1` fold whose materialized public meaning and semantic identity converge exactly with a clean full rebuild over the same canonical source state.

## Requirements
- [ ] Define a strict versioned source-change envelope for run, worker, work-item, project-memory, dossier, and pending-decision upsert/removal without inventing a second canonical database
- [ ] Apply exact duplicate changes idempotently, reject conflicting change-ID reuse and invalid source identities, and preserve deterministic ordering independent of arrival batches
- [ ] Incrementally recompute only affected source/entity relationships while sharing small facet/entity primitives with the full fold; do not implement the incremental path as a wrapper that simply calls `project_outcomes`
- [ ] Handle cross-entity effects such as finding acceptance/publication, proof/finding parent links, dangling-parent diagnostics, unknown states, degraded memory, and deletions
- [ ] Materialize the same public schema, entities, diagnostics, degraded flag, consistency guarantees, and W-0046 semantic SHA-256 as a full rebuild when canonical source state is equal
- [ ] Keep the engine pure and in-memory in this slice; production cache persistence, SSE adoption, and consumer migration remain separate

## Acceptance Criteria
- [ ] Differential tests apply deterministic randomized changes in many batch orders and compare the complete incremental projection and semantic identity with a clean full fold after every step
- [ ] Exact retries are no-ops, conflicting retries fail closed, and removal followed by re-add converges
- [ ] Finding decisions/dossiers and parent deletion/reappearance update every affected facet and diagnostic without stale cards
- [ ] Unknown future states remain preserved and degraded rather than rejected or normalized by guess
- [ ] Rebuilding from the accumulator's canonical source snapshot after process recreation yields byte-identical public JSON

## Plan
Refactor reusable entity/facet folds first, define one strict source-state/change boundary, then implement the independent incremental materializer and differential adversarial tests. Do not wire a production cache until parity is proved.

## Notes
Parity means identical public outcome meaning, not identical internal work performed. Observation watermarks remain non-identity. If derivation-path metadata must differ, keep it outside the semantic domain or expose a shared guarantee honestly; never label an incremental result `full-fold` merely to make hashes match.
