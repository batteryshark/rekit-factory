## Goal
Turn Mission Control's working per-launch safety choice and strategy selector into durable, named, inspectable policy inputs without weakening server-side authorization.

## Requirements
- [x] Define a versioned named safety-policy contract whose identity changes with every allowed-tool, approval, ceiling, or scope-binding change
- [x] Persist the selected policy identity and exact resolved policy on each run rather than only the browser's launch-time filtering choice
- [x] Expose richer strategy metadata including roles/graph shape, default ceilings, compatible profiles, and policy constraints from the server configuration
- [x] Render effective strategy and safety differences in the v3 composer before launch and in run detail after launch
- [x] Keep browser controls narrowing-only; crafted requests and stale policy IDs must still fail closed at the controller boundary

## Acceptance Criteria
- [x] Restarting Mission Control preserves named policies and a run remains bound to the exact policy revision it launched with
- [x] Two policies with the same label but different content cannot collide or silently replace one another
- [x] Strategy defaults are visible, deterministic, and validated by the controller rather than trusted from browser fields
- [x] Legacy launches remain readable with an explicit compatibility policy and no inferred permission expansion
- [x] Focused API, controller, UI-helper, and rendered composer tests pass without changing the canonical v3 design direction

## Plan
Define and test the immutable policy/strategy metadata contracts first, enforce them at launch/resume, then project the resolved values into the existing composer and run detail.

## Notes
The current Supervised and Automatic-only controls already work and W-0012 is complete. This task is a durability and metadata enhancement, not permission to add an authorization bypass or a second policy ledger.
