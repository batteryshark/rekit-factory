## Goal
Close W-0026's vocabulary gap by giving campaign coverage and archival state real canonical lifecycle sources, then projecting them without conflating covered, completed, successful, reported, or archived outcomes.

## Requirements
- [x] Define minimal durable campaign and archive source records/events with stable identities, allowed transitions, terminal semantics, and explicit transition authorities
- [x] Represent coverage as an orthogonal, versioned canonical outcome that cannot imply run completion, successful disposition, validation, acceptance, report publication, or archival
- [x] Project canonical `campaign` and `archive` entities, including parent relationships and degraded unknown/dangling handling, through the full outcome fold
- [x] Extend the strict source-change/source-snapshot contract and selective incremental fold for campaign and archive upsert/removal with exact full-fold parity
- [x] Version the public vocabulary honestly if coverage or archival semantics expand the existing v1 domain, retaining fail-closed handling for older and future projections
- [x] Expose the canonical campaign, coverage, and archive outcomes through the run API and Mission Control without independently deriving status in either consumer
- [x] Publish the complete transition/authority matrix for campaigns and archives alongside the existing entity vocabulary

## Acceptance Criteria
- [x] API and UI fixtures visibly distinguish `completed`, `covered`, `successful`, `reproduced`, `accepted`, `reported`, and `archived` without treating any pair as synonyms
- [x] A fully covered campaign with a running or failed run is not projected as completed or successful
- [x] A completed successful run with no durable archive source remains unarchived
- [x] A rendered report and a published proof bundle cannot create or advance archive state
- [x] Full and incremental projections converge after campaign/archive transition, removal, re-addition, unknown-state, and dangling-parent sequences
- [x] Campaign/archive source transitions name one authoritative owner and reject or degrade impossible history without rewriting canonical evidence
- [x] Mission Control renders campaign and archive entities from canonical projection bytes at desktop and narrow widths

## Plan
Specify and test the lifecycle/authority matrix first. Add the smallest durable source boundary, then extend the shared full and incremental projection primitives, API projection, and Mission Control fixtures. Keep coverage orthogonal even if that requires a vocabulary-version increment rather than overloading an existing facet.

## Notes
This slice does not claim crash injection at every event/artifact boundary, cross-store atomicity, production derived-cache recovery, or snapshot/dossier-list rebuild parity. Those remaining W-0026 acceptance gates require a separate task that first enumerates the actual durable boundaries and caches under test.
