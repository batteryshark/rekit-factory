## Goal
Turn the favored Mission Control mockup into the real operator surface over the control-plane API.

## Requirements
- [ ] Treat `docs/mockups/e7-mission-control-v3.html` as the canonical product and visual direction; convert its surfaces to live data incrementally instead of replacing it with a simplified dashboard
- [ ] Preserve the v3 information density, sidebar and view structure, cards, status language, detail tabs, tool rack, harness/profile surfaces, notifications, transitions, pulses, hover feedback, and reduced-motion behavior unless a deliberate reviewed change improves them
- [ ] Fleet cards show run, worker, model, coverage, elapsed time, and current-step state
- [ ] Decision Inbox supports allow, deny, missing-tool, and direction questions
- [ ] Project detail exposes activity logs, artifacts, worker reports, and model usage
- [ ] Activity detail can expose structured model lifecycle, thinking, tool-call, tool-result, retry, and validation events produced by W-0015 without leaking secrets
- [ ] New Run composes target, goal, tools, worker strategy, model profile, and safety policy

## Acceptance Criteria
- [ ] A user can launch, observe, answer, resume, and inspect the benign smoke investigation from the UI
- [ ] Every visible control is wired or visibly disabled
- [ ] A side-by-side interaction review against the v3 mockup confirms that live wiring has not flattened its visual hierarchy, motion, feedback, or breadth of interaction

## Plan
Continue the existing piece-at-a-time conversion of the v3 mockup. Keep each completed live slice visually and behaviorally aligned with v3 while W-0015 supplies the durable semantic event stream. Add the missing question types, worker reports and usage views, strategy/safety inputs, then run interaction and side-by-side visual audits over every visible control and view.

## Notes
`docs/mockups/e7-mission-control-v3.html` is the canonical design reference and should remain intact as the comparison fixture. The implementation may remove goalpack and retired-ledger concepts or adapt mock data to real Factory concepts, but should retain the mockup's character and interaction richness. New functionality should be composed into this design language rather than prompting a wholesale UI rewrite.
