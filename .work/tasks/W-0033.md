## Goal
Make long-running research understandable and steerable from Mission Control by exposing objectives, hypotheses, evidence, proof state, campaign health, budgets, and consequential decisions without requiring the operator to read raw model transcripts.

## Requirements
- [x] Extend the existing v3 design language rather than replacing it with a generic dashboard or separate writeup site
- [x] Add a project-memory view for current objective/pivot, active workstreams, prioritized next actions, open questions, decisions, failed attempts, and bounded session checkpoints
- [x] Add a hypothesis board grouped by proposed/testing/supported/disproved/reproduction state with confidence, expected/falsifying observations, evidence for/against, stop condition, and linked work
- [x] Add a finding/proof view that clearly separates candidate, demonstrated, reproduced, operator-accepted, rejected, withdrawn, and inconclusive states
- [x] Render proof manifests/dossiers, validation attempts, artifacts, redactions, prerequisites, and hash-verification state directly from canonical projections
- [x] Add campaign controls and health: epoch, elapsed, progress/novelty, coverage, cumulative/per-epoch budgets, retry/no-progress counters, scope revision, next checkpoint, pause/stop/resume, and reasoned terminal outcome
- [x] Add authorization/scope inspection and exact-content approval flows for scope or ceiling changes
- [x] Preserve the cross-project decision Inbox and add exception-notification delivery/acknowledgement state without making delivery a second inbox
- [ ] Show calibration/evaluation comparisons and promotion candidates only as links/projections owned by their respective systems
- [x] Keep raw semantic activity available for diagnostics but default the operator to structured research state and proof
- [ ] Ensure every badge/count/card is derived from the canonical outcome ontology and deep-links to its evidence
- [ ] Maintain keyboard, responsive/phone, reduced-motion, accessibility, visual-density, and two-second needs-you scan goals

## Delivered Partial UI Slices
- [x] W-0012 exposes canonical project memory in the e7 run-detail language: current goal, workstreams, attempts/negative results, decisions, theories, open questions, next actions, diagnostics, evidence references, and bounded resume context
- [x] W-0023 exposes safe evidence metadata and retention lifecycle controls without disclosing quarantined bytes or private evidence paths
- [x] W-0025 exposes contained proof-dossier list/open/export views linked from the canonical run ledger
- [x] W-0048 exposes a live Outcomes workspace with distinct execution, completion, disposition, validation, acceptance, and publication facets, canonical links, integrity/degradation feedback, keyboard filtering, responsive density, and reduced-motion behavior
- [x] The first W-0033 workspace slice joins canonical hypothesis/test/observation and finding/reproduction/decision projections into lifecycle boards inside the existing e7 Memory surface, with exact evidence/outcome navigation and no transcript-derived state
- [x] Campaign Watch provides a one-screen cross-campaign operator scan for needs-action ordering, canonical coverage/novelty, remaining work/cost/time authority, and the strongest exactly linked reproduced or accepted result
- [x] Finding cards provide bounded unified proof detail from canonical finding, dossier, artifact, and evidence projections: proof prerequisites, reproduction/validation attempts, environment differences, manifest/finding-state hashes, dossier verification, artifact hashes, redaction/truncation/restriction state, and exact proof/evidence/artifact links
- [x] Campaign pause/resume/stop and scope/ceiling change decisions use canonical campaign revisions and durable exact operation identities; the API rejects extra browser-owned fields, stale revisions, and conflicting replays
- [x] Notification controls retain and independently qualify exact run/entity identity, translate finding links onto the canonical Outcomes surface, and focus the exact decision, finding, or proof record instead of opening only a coarse tab

These checks record delivered component slices, not completion of this epic. Direct research controls, authorization approval flows, calibration/promotion links, exhaustive cross-surface navigation, and full workspace acceptance remain open.

## Acceptance Criteria
- [x] From one screen, the operator can identify which campaign needs action, what it is testing, whether progress is real, remaining budget, and strongest reproduced result
- [ ] An operator can inspect contradictory evidence, stop a bad workstream, approve/deny a scope or finding decision, and resume without touching storage or a terminal
- [ ] Candidate and reproduced findings are visually unmistakable and no stale report/index state appears after crash/reconnect
- [ ] Deep links from desktop/webhook notifications open the exact durable decision, finding, proof, or terminal outcome
- [ ] A side-by-side review confirms the new surfaces retain Mission Control v3's hierarchy, density, motion language, and interaction quality
- [ ] Browser tests cover concurrent updates, reconnect, empty/degraded states, large histories, redactions, keyboard access, mobile layout, and reduced motion
- [ ] No UI control can expand scope, raise a hard ceiling, validate a finding, or publish a capability without the required durable decision

## Plan
Design the information architecture over real read models, add read-only hypothesis/memory/proof panels incrementally, then campaign controls and decision flows, then calibration/promotion links and exhaustive interaction/visual QA.

## Notes
This task begins after W-0012 establishes the live v3 baseline, but individual read-only panels may land as their source contracts stabilize. The UI is a projection and control surface; it does not own research truth.
