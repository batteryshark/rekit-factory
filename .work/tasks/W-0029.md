## Goal
Convert repeated, independently verified research successes into reviewable capability improvements without allowing an active worker to modify its own tools, policies, prompts, or canonical knowledge.

## Requirements
- [x] Define immutable promotion candidates with origin runs, proof bundles, fixture/evaluation results, proposed capability kind, scope, prerequisites, risks, version, and content hash
- [ ] Classify each candidate correctly as reusable knowledge, a behavioral skill, deterministic tool/adapter, strategy/policy change, benchmark fixture, or target-specific evidence that must not be promoted
- [ ] Require repeated success across more than one development fixture and a configured held-out evaluation before a candidate is eligible
- [ ] Measure regression, false-positive, intervention, reproducibility, cost, platform coverage, and safety-policy behavior; do not gate solely on solve rate
- [ ] Search existing Rekit capabilities and re-kb concepts before proposing creation; prefer enrichment/fix over duplicates
- [ ] Stage every proposed change in an isolated branch/worktree/package with exact dependencies and no mutation to the active run or installed capability roots
- [ ] Run appropriate deterministic tests, ReverseBench evaluations, security/safety checks, documentation validation, and OKF validation where applicable
- [ ] Require explicit operator review and approval for staging and a separate decision for installation/merge/publish
- [ ] Bind approval to candidate content, evidence set, evaluation version/results, base revision, and resulting diff; any change invalidates approval
- [ ] Support rejection, quarantine, rollback, deprecation, and supersession while preserving provenance and prior measurements
- [ ] Never promote raw exploit payloads, credentials, private target logic, personal data, or target-specific chains as reusable capabilities

## Acceptance Criteria
- [ ] A technique that succeeds on development fixtures but fails held-out evaluation is rejected with its evidence intact and no installed changes
- [ ] A genuinely general technique produces the appropriate staged skill/knowledge/tool candidate with tests and provenance
- [ ] The same candidate is deduplicated across concurrent originating campaigns
- [ ] Editing a staged candidate after approval makes the approval stale and blocks installation
- [ ] Denial or failed validation leaves active Rekit/re-kb roots byte-for-byte unchanged
- [ ] Installing an approved candidate cannot alter or retroactively affect the run that proposed it
- [ ] Rollback restores the previous capability version while retaining audit and evaluation records

## Plan
Build classification/candidate records and eligibility policy first. Connect ReverseBench and re-kb dedupe second. Add isolated staging and dual approval last, using existing W-0018 decisions rather than inventing an alternate knowledge write path.

## Notes
This is the disciplined version of “capture a proven capability as a skill.” The promotion subject is observed generalization, not a model's confidence or one successful challenge.
