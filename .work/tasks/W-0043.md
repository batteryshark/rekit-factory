## Goal
Persist canonical v1 trial records locally through an API-enforced append-only store so exact retries are idempotent, conflicting trial-ID reuse fails closed, and committed records survive restart and interrupted writes unchanged inside a trusted local directory.

## Requirements
- [x] Provide append, read, and deterministic list APIs over canonical TrialRecordV1 bytes
- [x] Publish atomically and durably without exposing a partial canonical record as committed
- [x] Treat an exact same-ID/same-bytes retry as success and any same-ID/different-content retry as a conflict that preserves the first record
- [x] Enforce the same invariant across concurrent writers and process restart
- [x] Verify record identity, canonical bytes, safe storage identity, and content digest on every read/list operation
- [x] Fail closed on incoherently corrupted, truncated, noncanonical, mismatched, or stably symlinked records without silently replacing a committed record; state explicitly that coherent hostile local rewriting requires an external trust anchor

## Acceptance Criteria
- [x] Exact retry before and after reopening is idempotent; conflicting reuse leaves the original byte-identical
- [x] Failure before publication leaves no readable trial; failure after publication cannot roll back or replace it
- [x] Concurrent exact writers converge and concurrent conflicting writers produce one valid winner plus one closed conflict
- [x] Multiple completed and interrupted trials survive restart and list in deterministic order
- [x] Incoherent corruption, truncation, noncanonical bytes, mismatched identity/digest/schema, unsupported versions, and stable database/parent/sidecar symlink replacement are rejected

## Plan
Implement storage and focused temporary-directory tests in an isolated ReverseBench worktree after one worker slot opens. Shared exports/default paths remain an integration-owner decision.

## Notes
This is local durable record persistence only: no scheduler orchestration, Factory journal access, mutable lifecycle projection, distributed database, remote/object storage, retention, encryption, or multi-host consensus. Append-only is an API and verified-schema guarantee inside a trusted local directory. A malicious local writer able to coherently replace SQLite rows, digests, schema, and guards cannot be detected without an external anchor or signing key; this task does not pretend otherwise.
