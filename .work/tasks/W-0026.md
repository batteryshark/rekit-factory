## Goal
Eliminate ambiguous completed/solved/validated/indexed states and synchronization races by defining one outcome vocabulary and deriving all views from canonical committed events and artifacts.

## Requirements
- [ ] Publish state machines and terminal/non-terminal semantics for campaigns, runs, work items, workers, hypotheses, findings, validations, proof bundles, reports, and operator decisions
- [ ] Define which authority owns each transition: Muster, Factory scheduler, validator policy, Rekit tool result, or operator
- [x] Prevent a child entity from implying a parent outcome; for example, worker done does not mean run successful, and report rendered does not mean finding validated
- [x] Make Mission Control, API snapshots, exports, notification triggers, and reports pure/tolerant projections from canonical state
- [x] Use stable event cursors and transactional commit order so an outcome is never externally visible before its required proof/artifact references exist
- [x] Add deterministic full rebuild and incremental projection paths that converge to identical state
- [x] Detect impossible transitions, dangling references, stale derived data, and partial commits; expose repairable degraded state instead of guessing
- [x] Version public projection schemas and retain forward-compatible unknown state handling
- [x] Define latency/consistency guarantees for live SSE clients and reconnect behavior

## Acceptance Criteria
- [ ] Completed, covered, successful, reproduced, accepted, reported, and archived are distinguishable in API and UI fixtures
- [ ] Crash injection at every event/artifact boundary never produces a validated card without proof or a proof invisible after commit
- [ ] Rebuilding from canonical state after deleting derived caches yields the same snapshot and dossier list
- [x] Two concurrent clients and an SSE reconnect converge without duplicate cards or lost terminal transitions
- [x] Invalid/stale historical states are surfaced with diagnostics and do not silently mutate canonical evidence
- [x] The transcript's completed-versus-solved and writeup-index race classes are covered by regression tests

## Plan
Write the outcome glossary/transition matrix first, then centralize folds and projection schemas, then transactional publication and crash tests. Migrate each consumer incrementally and remove independent status/index mutation once parity is verified.

## Notes
This task is not a new database. It is an invariant layer over Muster/Factory events, artifacts, and read models.
