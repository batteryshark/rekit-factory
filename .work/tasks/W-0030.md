## Goal
Notify the operator only when durable state materially changes or action is required, with reliable deduplication, redaction, retry, acknowledgement, and deep links back to Mission Control.

## Requirements
- [x] Define a transactional notification outbox derived from committed canonical transitions, never ad-hoc model messages
- [x] Give every notification a stable dedupe key based on run/entity/transition/policy revision and an immutable redacted payload
- [ ] Notify on configurable consequential events: needs authorization/direction, high-confidence reproduction, budget/risk threshold, terminal campaign outcome, infrastructure failure requiring action, and stale operator decision
- [x] Suppress routine progress chatter, repeated still-working updates, streaming deltas, duplicate solve/writeup events, and transitions that immediately self-resolve
- [x] Support immediate, batched, digest, muted, escalation, and quiet-hours policies per project/campaign/severity
- [x] Provide best-effort desktop delivery plus a generic authenticated webhook adapter suitable for Slack/Pushover/n8n-style consumers without making any external service authoritative
- [x] Store credential references outside run state; redact target secrets, private paths, raw evidence, tokens, and sensitive findings from payloads
- [x] Retry with bounded exponential backoff and idempotency; delivery failure never blocks canonical run progress unless an explicit policy says human acknowledgement is required
- [x] Track queued/sent/failed/acknowledged/superseded state and link to exact Mission Control entity/decision
- [ ] Let the operator test a channel and preview the exact redacted payload before enabling it

## Delivered Partial UI Slices
- [x] Mission Control raises one generic, redacted operator-attention notification when a run's fleet projection transitions from no waiting decisions to one or more waiting decisions
- [x] Initial fleet hydration and unchanged SSE reconnects do not re-alert, and the notice supports Open Inbox, Later, explicit dismiss, Escape dismissal, responsive placement, and reduced motion

These checks record the already-delivered W-0012 presentation slice only. Its bounded in-memory deduplication is not the durable transactional outbox, delivery state, acknowledgement, retry, channel, digest, or quiet-hours system required to complete W-0030.

## Acceptance Criteria
- [x] Replaying or reconnecting to duplicate events produces one delivered notification for the same transition
- [ ] A run that emits many progress events and one permission decision produces exactly one actionable alert
- [ ] A finding notification is sent only at the configured reproduced/accepted threshold and links to its proof dossier
- [x] Secrets injected into every source field do not appear in payload, logs, URLs, or delivery errors
- [x] An unavailable receiver retries within limits, records failure, and leaves the run/decision safely available in Mission Control
- [x] Digest and quiet-hours behavior is deterministic under a fake clock and survives restart

## Plan
Implement the pure transition-to-notification policy and durable outbox first, then desktop/webhook adapters, then preferences, acknowledgement, digests, and Mission Control configuration.
