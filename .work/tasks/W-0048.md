## Goal
Bring the canonical outcome ontology into the favored e7 Mission Control experience so operators can distinguish execution, completion, disposition, validation, acceptance, and publication without flattening the interface into a generic status table.

## Requirements
- [x] Add a first-class live Outcomes surface inside the existing v3 run-detail hierarchy, preserving its density, layered cards, motion/feedback language, keyboard behavior, responsive layout, and reduced-motion policy
- [x] Render only canonical `outcomeProjection` entities/facets/diagnostics with clear type, parent, owner, known/terminal, degraded, and unknown-state treatment; do not infer parent success or invent findings
- [x] Make run, work, worker, finding, validation, proof, and operator-decision outcome classes visually distinguishable and scannable in roughly two seconds
- [x] Use `semanticSha256` to avoid semantic rerenders on watermark-only updates, verify the digest client-side or surface a bounded degraded warning, and remain tolerant of older snapshots without the field
- [x] Provide accessible type/state filters, search, entity counts, empty/degraded states, focus management, and links to existing evidence/activity/report/decision surfaces where the canonical data supports them
- [x] Keep raw private paths, evidence bodies, prompts, secrets, and notification payload details out of the projection cards

## Acceptance Criteria
- [x] A populated benign run clearly shows completed versus successful versus validated/accepted/published as separate facets
- [x] A child terminal state never visually promotes its parent and unknown/dangling states remain visibly degraded
- [x] Watermark-only SSE updates retain DOM/focus state while a semantic digest change updates the workspace exactly once
- [x] Keyboard, narrow/mobile, reduced-motion, empty, large-history, and diagnostic states have deterministic tests
- [x] A rendered comparison against `docs/mockups/e7-mission-control-v3.html` confirms the new surface extends rather than dilutes the chosen direction

## Plan
Add the tab/surface using the existing packaged HTML/CSS/JS architecture, create pure projection-to-view helpers and focused DOM-independent tests, then perform a live desktop/narrow/reduced-motion review before integration.

## Notes
Mission Control remains a projection/control surface, not an outcome authority. W-0047 incremental parity and later consumer migration may optimize how updates arrive, but this slice must work correctly from complete W-0046 snapshots now.
