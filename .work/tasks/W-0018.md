## Goal
Turn durable investigation findings into reviewable re-kb contribution candidates without treating raw logs, model output, or unverified theories as publishable knowledge.

## Requirements
- [x] Candidate extraction separates durable reusable knowledge from target-specific evidence, behavioral instructions, code, credentials, and unsafe material
- [x] Deduplication searches existing concepts before proposing creation or enrichment
- [x] Every proposal distinguishes observation from interpretation and carries run/artifact/event provenance
- [x] The design requires explicit operator decisions and isolated staging rather than modifying re-kb during an investigation
- [x] Staging plans require concept, parent-index, newest-first log, and strict OKF validation updates; filesystem/Git execution belongs to the later promotion pipeline

## Acceptance Criteria
- [x] A fixture run produces a reviewable proposal but no knowledge-base mutation before approval
- [x] Approval creates an immutable, mutation-free staging plan; denial leaves the bundle untouched
- [x] Secrets, raw binaries, personal data, persistence payloads, and weaponized chains are rejected before staging

## Plan
Specify the candidate/provenance schema and review lifecycle first. Integrate with re-kb `INGEST.md` and strict validation only after read-only retrieval is stable.

## Notes
Logging and knowledge publication are different lifecycles. Logs remain append-only run evidence; only distilled, deduplicated, reviewed concepts enter re-kb.
