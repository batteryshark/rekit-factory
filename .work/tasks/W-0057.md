## Goal
Make initial subscription, reconnect, and stale/foreign cursor recovery converge each client to one current run snapshot without replay storms, duplicate cards, or lost terminal transitions.

## Requirements
- [x] Seed a new run EventSource from the latest event already present in the fetched snapshot instead of replaying the complete history
- [x] Preserve the latest acknowledged event ID when Mission Control deliberately recreates a stream after an error
- [x] Distinguish initial subscription, known cursor continuation, and unknown/stale cursor recovery at the server boundary
- [x] Recover an unknown cursor with one explicit reset signal rather than replaying every historical event or silently returning no state
- [x] Keep cursor IDs run-scoped and prevent a cursor from another run from skipping events
- [x] Retain W-0048's old-stream identity guards so delayed messages/errors cannot affect the replacement stream

## Acceptance Criteria
- [x] Two clients starting from the same snapshot receive each later event once and converge to the same semantic projection
- [x] Reconnect after a known event yields only later events
- [x] A stale or foreign event ID causes one bounded reset/refresh and then normal continuation
- [x] Opening a completed run does not issue one snapshot refresh per historical event
- [x] Terminal events are neither lost nor duplicated across the disconnect boundary

## Plan
Define a pure cursor-batch decision first, cover initial/known/unknown/cross-run cases, then thread the latest snapshot/event ID through Mission Control's guarded EventSource lifecycle and exercise two-client reconnect behavior against the loopback API.

## Notes
Event IDs remain transport cursors, not outcome semantic identity or an ETag. Clients still replace the complete canonical snapshot after a signal; this task only makes signal delivery bounded and restart-safe.
