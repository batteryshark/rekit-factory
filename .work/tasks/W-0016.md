## Goal
Let Factory discover and safely invoke capabilities from multiple configured roots, including public Rekit and private IDA/debugger tooling, without copying private manifests into the public repository.

## Requirements
- [x] Configuration accepts ordered, repeatable capability roots and exposes their health/origin without persisting secrets
- [x] Compatible `registry.json` + dispatcher roots compose into one catalog while every tool retains its owning dispatcher
- [x] Duplicate IDs fail closed unless an explicit, inspectable precedence/namespace rule resolves them
- [x] Private standard-skill and `mcp-registry.json` sources are classified honestly; dispatchable private capabilities are migrated behind the compatible root contract while MCP catalogs remain a separate source type
- [x] Mission Control groups or labels capabilities by source and never leaks private filesystem paths to run records or browser clients

## Acceptance Criteria
- [x] A fixture with two compatible Rekit roots lists and invokes one tool from each through the correct dispatcher
- [x] Collision, missing-root, unhealthy-dispatcher, and private-path-redaction tests pass
- [x] The current `rekit-private` IDA/other capability layout has a documented adapter or migration path

## Plan
Land the federated compatible-Rekit catalog first, then add typed adapters for private standard skills and MCP servers rather than weakening the invocation contract.

## Notes
The current public checkout has `registry.json` and `bin/rekit`. As of this audit, `rekit-private` has standard skill folders plus `mcp-registry.json`, but no root `registry.json`/`bin/rekit`; those are distinct source types.
